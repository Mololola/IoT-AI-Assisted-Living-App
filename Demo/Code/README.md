# Night Wandering Detection — Bench Demo

Álvaro Menéndez · IoT-AI Assisted Living · March 2026

---

## Overview

This is a simplified demo of our assisted living system, focused on one scenario: **night wandering detection in the bedroom**.

We have 4 sensors in the bedroom and 3 LED strips in the hallway. The Raspberry Pi monitors the sensors via MQTT, and when it determines the elderly person has left their bed at night (bed empty, nobody detected in the room, lights off), it turns on the hallway LEDs to light their path and pushes a HIGH-severity alert to the cloud API, which the caregiver sees on the mobile app.

The system uses two detection layers:
- A **rule engine** that checks conditions directly (works from second one, no training needed)
- An **ML engine** (Isolation Forest) trained on what "normal sleep" looks like, which flags anything unusual

Both run simultaneously. The rule engine is fast and deterministic; the ML engine catches subtler patterns.

## Hardware

**Sensors (4 ESP32 boards in the bedroom):**

| Sensor | Topic | What it sends |
|--------|-------|---------------|
| Bed pressure (FSR) | `home/bedroom/pressure` | 0–5000 (integer, ADC mapped). Above 500 means someone is in bed |
| PIR motion | `home/bedroom/PIR` | "IN" or "OUT" — toggles each time motion is detected |
| mmWave presence | `home/bedroom/presence` | Distance in cm. Under 50cm means someone is close |
| Light | `home/bedroom/light` | "Day" or "Night" |

**Actuators (3 ESP32 boards in the hallway):**

| Actuator | Topic | What it receives |
|----------|-------|-----------------|
| 3x LED strips | `home/hallway/led` | "ON" or "OFF". All three subscribe to the same topic so they activate together |

All communication goes through the **Mosquitto MQTT broker** running on the Pi.

## How to run it

### 1. Install dependencies (once)

```bash
pip install paho-mqtt scikit-learn numpy joblib requests --break-system-packages
```

### 2. Train the model (before demo day, no sensors needed)

```bash
cd ~/demo
python3 demo_train.py
```

This generates synthetic "normal night" data (person sleeping, occasional bathroom trips, etc.) and trains an Isolation Forest on it. Takes about 30 seconds. The model files end up in `models/`.

### 3. Verify hardware (on demo day, after powering on ESP32s)

```bash
python3 demo_check.py
```

Listens for 20 seconds and reports which sensors are sending data. Also flashes the LEDs to confirm they respond.

### 4. Run the demo

```bash
python3 demo_run.py
```

## What happens when you run it

The script uses a **simulated clock** — it doesn't use the real time of day (otherwise we'd need to demo at 2 AM). Instead, it starts at 22:00 and runs at 60x speed, so 1 real minute = 1 simulated hour. The simulated time is shown on screen.

The terminal output looks like this:

```
  SIM TIME   | Real    |      Bed    Pres  PIR  Light | LEDs | Status
  ==============================================================
   10:00 PM  |  0m 00s | OCCUPIED    NEAR   IN  NIGHT |   -  | .. Sleeping
   11:00 PM  |  1m 00s | OCCUPIED    NEAR   IN  NIGHT |   -  | .. Sleeping
   12:00 AM  |  2m 00s | OCCUPIED    NEAR   IN  NIGHT |   -  | .. Sleeping
    1:00 AM  |  3m 00s | OCCUPIED    NEAR   IN  NIGHT |   -  | .. Sleeping
```

After about 4 real minutes the clock reaches 02:00 AM, and you can trigger the anomaly.

**On startup**, the script also pushes data to Daniel's cloud API:
- Creates a resident profile
- Uploads the trained ML model

So the Flutter app has something to display from the start.

### Triggering night wandering

To simulate the person leaving the bed:

1. Remove the weight from the pressure sensor (bed becomes "empty")
2. Trigger the PIR so it toggles to "OUT"
3. Step away from the mmWave sensor (more than 2 metres)
4. Make sure the light sensor reads "Night" (cover it if needed)

When all four conditions are met, the system detects night wandering:

```
    2:15 AM  |  4m 15s |    EMPTY  ABSENT  OUT  NIGHT |  ON  | !! WANDERING
```

The hallway LEDs turn on, and two alerts are pushed to the cloud — one from the rule engine, one from the ML engine. Both are HIGH severity.

### Person returns

Put weight back on the pressure sensor and/or move close to the mmWave sensor. The LEDs stay on for 5 more seconds (grace period to avoid flickering), then turn off.

### Speed options

```bash
python3 demo_run.py --start-hour 1     # start at 1 AM, can trigger immediately
python3 demo_run.py --speed 120        # faster: reaches 2 AM in ~2 minutes
python3 demo_run.py --speed 30         # slower: more time to explain to the supervisor
python3 demo_run.py --no-cloud         # skip cloud API (if there's no internet)
```

### Stopping

Press Ctrl+C. It prints a summary of alerts, LED events, and simulated time covered.

## Detection logic

### Rule engine (deterministic, always active)

Fires when ALL of these are true at the same time:
- Pressure < 500 (bed empty)
- Presence > 200 cm (nobody in the room)
- PIR = OUT (no recent motion)
- Light = Night (it's dark)
- Simulated time is between 22:00 and 07:00

### ML engine (Isolation Forest, needs training)

The model was trained on what a normal night looks like: high bed pressure, nearby presence reading, low PIR activity, lights off. When the live sensor readings deviate significantly from that pattern (like the bed being empty for a long time at 2 AM), the model scores it as anomalous.

We extract 12 features per sliding window (bed occupancy rate, pressure variation, presence distance stats, PIR activity, light state, etc.) and feed them to the Isolation Forest. Contamination is set to 3%, meaning roughly 3% of normal data is expected to be slightly unusual.

## Cloud API integration

Every alert gets pushed to Daniel's backend at `https://assisted-living-platform.onrender.com`.

The alert format:

```json
{
  "alert_id": "RULE_NIGHT_WANDERING_20260305_021500",
  "timestamp": "2026-03-05T14:15:00.000Z",
  "rule_type": "NIGHT_WANDERING",
  "severity": "HIGH",
  "message": "Person out of bed at 2:15 AM - bed empty, not in room, dark",
  "recommended_action": "Check on resident - possible disorientation",
  "source": "RULE_ENGINE",
  "anomaly_score": null,
  "sensor_data": {
    "bed_pressure": 45,
    "presence_cm": 350,
    "pir_state": "OUT",
    "light_state": "NIGHT",
    "simulated_time": "02:15"
  }
}
```

Relevant endpoints for Luis's app:
- `GET /alerts?severity=HIGH` — urgent alerts
- `GET /alerts?acknowledged=false` — unacknowledged alerts
- `PUT /alerts/{id}/acknowledge` — mark alert as handled
- `GET /models/latest` — check if model is uploaded
- `GET /residents` — resident profile

## Troubleshooting

**MQTT connection fails** — run `sudo systemctl start mosquitto`.

**Sensor not showing up** — check that the ESP32 is powered, connected to the right WiFi network, and that the serial monitor shows it connecting to the broker.

**LEDs don't react** — verify the LED ESP32s are subscribing to `home/hallway/led` (shared topic). Check the serial output for "Subscribed to command topic."

**ML doesn't detect anything** — make sure you ran `demo_train.py` first. If you changed thresholds in the config, retrain.

**Cloud push fails** — Render's free tier goes to sleep after inactivity. The first request might take 30–60 seconds to wake it up. The script handles this gracefully and logs "cloud: offline" if it can't reach the API.

**PIR doesn't toggle** — there's a 3-second debounce built into Mía's code. Wait between triggers.

## Tuning thresholds

If the sensors give values outside the expected ranges, edit `demo_config.json`:

```json
"thresholds": {
    "pressure_occupied": 500,
    "pressure_empty": 100,
    "presence_near_cm": 50,
    "presence_far_cm": 200
}
```

If you change these, retrain the model (`python3 demo_train.py`) since the feature extraction uses the same thresholds.
